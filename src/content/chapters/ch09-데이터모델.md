## 1. 개요 -- 데이터 모델 설계 원칙

### 1.1 설계 목표

| 원칙 | 설명 |
|------|------|
| **합집합 커버리지** | 4개 소스의 모든 필드를 누락 없이 수용한다 |
| **타입 안전성** | VB6의 `Currency`/`String`, EDB의 `string`-저장-`decimal`-변환 패턴을 정리하여 타입을 명확히 한다 |
| **불변 입력 / 가변 출력** | CalcOptions + DrugItem은 계산 중 불변, PrsBillM + CalcResult는 계산 엔진이 채운다 |
| **마스터 분리** | 수가/요율/산정특례 마스터는 Repository 패턴으로 분리하여 DB/메모리 교체 가능하게 한다 |
| **공단 전자문서 정합성** | 최종 출력 필드가 공단 명세서 일반내역(p147-151) + 조제투약내역(p152-156)의 항목과 1:1 매핑되어야 한다 |

### 1.2 소스 약칭

| 약칭 | 소스 | 언어 | 특징 |
|------|------|------|------|
| **BIZ** | 비즈팜 `Ppre1000__.frm` | VB6 | UI-중심, Currency 타입, Z코드 직접 조회 |
| **HIRA** | 공단 전자문서 작성요령 p147-156 | 텍스트 | 전자문서 규격, an/n 타입, POSITION 기반 |
| **UPH** | 유팜 `Upharm2016.Model` | C# | ORM-기반, `ST조제료`/`ST조제료적용일` 엔티티 |
| **EDB** | EDB `NewEPharm.CostCalc` | C# | 클린 아키텍처, PascalCase, record struct 사용 |

### 1.3 데이터 흐름 개요

```
CalcOptions + DrugItem[]       (입력)
       │
       ▼
 ┌─────────────┐    조회     ┌──────────────┐
 │ CalcEngine   │──────────▶│ Repository    │
 │ (계산 로직)   │◀──────────│ (마스터 조회)  │
 └──────┬──────┘   반환     └──────────────┘
        │                    - MediWageM (수가)
        │                    - InsuRateM (요율)
        │                    - MediIllnessM (특례)
        ▼
 PrsWageListM[]              (중간: 수가 리스트)
        │
        ▼
 PrsBillM + CalcResult       (출력)
        │
        ▼
 공단 전자문서 변환            (최종 매핑)
```

### 3.2 PD_SUM 자동계산 공식

```
PD_SUM 계산 (EDB DrugItem.RecalcSum):
  amount = PD_DOSE * PD_DNUM * PD_DDAY
  if PD_PACK > 0: amount = amount / PD_PACK
  PD_SUM = (int)(amount * PD_PRICE + 0.5)    // 원 미만 4사5입
```

**HIRA 규격 대조**: `금액 = 단가 * 1회투약량 * 1일투여량 * 총투여일수` (원미만 4사5입) -- 일치.

### 3.3 BIZ 대응

비즈팜은 스프레드시트(sprDrug) 열 기반으로 약품을 관리:

| sprDrug Col | 대응 필드 |
|-------------|-----------|
| Col=1 | TypeButtonText ("보험"/"비"/"처방") → PD_INSUPAY 변환 |
| Col=4 | 단가 → PD_PRICE |
| Col=5 | 1회투약량 → PD_DOSE |
| Col=6 | 1일횟수 → PD_DNUM |
| Col=7 | 투약일수 → PD_DDAY |
| Col=8 | 금액 → PD_SUM |
| Col=18 | 내/외/주사 구분 → PD_TAKE |

## 5. 마스터 모델: 보험료율 (InsuRateM)

> 보험코드별 본인부담률을 관리. PK: 적용일자 + 보험코드.

### 5.1 필드 목록 (11개)

| # | 필드명 | 타입 | DB 컬럼 | 설명 |
|---|--------|------|---------|------|
| 1 | `DueDate` | string | IR_DUEDATE | 적용 기준일자 (PK1) |
| 2 | `InsuCode` | string | IR_INSUCODE | 보험 구분 코드 (PK2, 예: C10, D10, G10) |
| 3 | `Rate` | string→decimal | IR_RATE | 기본 본인부담률(%). 100=전액본인, 0=부담없음 |
| 4 | `SixAgeRate` | string→decimal | IR_6AGE_RATE | 6세 미만 경감 비율 |
| 5 | `FixCost` | decimal | IR_FIX_COST | 정액 본인부담금 |
| 6 | `HealthCenter` | decimal | IR_HEALTH_CENTER | 보건기관 금액 |
| 7 | `Mcode` | decimal | IR_MCODE | 의료급여 M코드 수급권자 정액 |
| 8 | `Bcode` | decimal | IR_BCODE | 의료급여 B코드 수급권자 정액 (하한 보호) |
| 9 | `V2520` | string→decimal | IR_V2520 | 중증질환 등급 0/4 본인부담률 |
| 10 | `V2521` | string→decimal | IR_V2521 | 중증질환 등급 1 본인부담률 |
| 11 | `Age65_12000Less` | string→decimal | IR_65AGE_12000_LESS | 65세 이상 12,000원 이하 경감률 |

### 5.2 적용 조건 상세

| 필드 | 적용 조건 |
|------|-----------|
| `Rate` | 기본 본인부담률. 모든 케이스의 기본값 |
| `SixAgeRate` | 6세 미만 AND 전액부담 아님 AND 부담률 10% 초과 AND 값 존재 AND 값 != "0" → `insuRate *= SixAgeRate / 100` |
| `FixCost` | 건보C (65세+, 10,000이하), 의료급여D, 기타보험G 시 정액 적용 |
| `HealthCenter` | 보건기관 여부에 따른 고정 금액 |
| `Mcode` | D타입 + V252 중증 sbrdnType="M" 또는 D10 일반 sbrdnType="M" |
| `Bcode` | D타입 sbrdnType="B001/B002/B003/B007/B009" 또는 D10 sbrdnType="B" |
| `V2520` | V252 질병코드 AND 등급 "0"/"4" |
| `V2521` | V252 질병코드 AND 등급 "1" |
| `Age65_12000Less` | 65세 이상 AND Extype_9 차감 후 합계 12,000원 이하 AND 조제일자 >= 20180101 AND 특례질환 없음 |

### 5.3 BIZ 대응

비즈팜은 `SelfRate()` 함수에서 보험유형별 본인부담률을 직접 분기:
- `Percent_GB`: "1"=정액, "2"=정률
- `intZBaseAmt`: PZCVALUE의 기본금액 (= FixCost 대응)
- `intZBase`: 정액/정률 기준 구간 (65세 이상 구간 판별)

### 5.4 UPH 대응

유팜은 `ST조제료적용일`에 본인부담 관련 필드를 내장:
- `D정률`, `D정액1~3`: EDB의 Rate, FixCost 대응
- `D보호1종~8종`, `P보호1~8`: 의료급여 종별 정액 (Mcode/Bcode 대응)
- `소아정률`: SixAgeRate 대응

### 5.5 통합 설계 권고

```

class CalcResultWage {
    // 기본조제료 구성
    decimal PharmMgmtFee        // 약국관리료 (Z1000)
    decimal BasicPrepFee        // 기본조제기술료 (Z2000)
    decimal MedGuide            // 복약지도료 (Z3000)
    decimal DrugMgmtFee         // 의약품관리료 (Z5000)
    decimal NightFee            // 야간조제관리료
    decimal NonFaceToFaceFee    // 비대면조제관리료

    // 약품조제료 구성 (Z4000)

                        + DrugMgmtFee + NightFee + NonFaceToFaceFee
    decimal DrugPrepTotal => InternalFee + InternalPowderFee
                           + ExternalFee + InjectionFee + DualFee
    decimal GrandTotal => BasicTotal + DrugPrepTotal
    decimal GrandTotalTruncated   // 절사 옵션 반영 후 최종값
```

### 13.6 주의사항

1. **EDB의 string 저장 패턴**: `InsuRateM`의 `Rate`, `SixAgeRate`, `V2520`, `V2521`, `Age65_12000Less`는 DB에서 string으로 저장되나 계산 시 decimal 변환이 필요. 통합 설계에서는 decimal로 통일하되, DB 어댑터에서 변환 처리.

2. **MediWageM의 탭 구분 문자열**: EDB의 비정규화 구조(탭 구분 1행)는 성능 최적화이나 유지보수 난이도가 높음. 통합 설계에서는 딕셔너리 구축 패턴은 유지하되, DB 저장 방식은 별도 결정.

3. **유팜 ST조제료적용일의 48개 필드**: 본인부담/소아/야간 등 계산 파라미터가 마스터 테이블에 내장. EDB는 이를 InsuRateM + CalcOptions로 분리. 통합 설계에서는 EDB식 분리를 권장하되, 유팜의 풍부한 파라미터를 누락하지 않도록 SugaParameters 객체로 보완.

4. **비즈팜의 UI-로직 결합**: `pnl_TotalSelfAmt`의 gSelfAmt 반올림, `txtRealAmt` 우선 적용, `txtCredit` 차감 등 UI 레이어 로직이 계산에 영향. 통합 설계에서는 이를 CalcOptions의 옵션 플래그로 분리.

5. **공단 전자문서 필드 갭**: HIRA 명세서 일반내역에 `지원금 n(10)`, `보훈본인일부부담금 n(10)` 등 PrsBillM에 직접 대응 필드가 없는 항목이 존재. 통합 설계에서 추가 필드로 보완 필요.
