# CH08. 특수케이스/엣지케이스 통합

> **소스**: EDB(`PrsBillCalcM.*`), 유팜(`보훈국비100.cs`), 비즈팜(`Ppre1000__.frm`), 공단 PDF(p.648~655), `docs/약국_약제비_계산_분석.md`
> **범위**: 명절가산, 본인부담상한제, 특수약품 하드코딩, 보훈병원 하드코딩, 날짜 분기, 비급여 반올림, U항 처리, 비즈팜 고유 버그, 반환/취소/수정처방, 소스별 비교

## 1. 개요

약제비 계산에는 일반 산정 로직으로 처리할 수 없는 **특수케이스**(정책 기반 예외)와 **엣지케이스**(경계값/결함 기반 예외)가 다수 존재한다. 이 문서는 4개 소스(EDB, 유팜, 비즈팜, 공단 PDF)에서 확인된 특수케이스를 통합 정리하며, 새 시스템 구현 시 반드시 반영해야 할 사항을 명시한다.

**핵심 주의사항**: 비즈팜(VB6)에서 발견된 버그는 새 시스템에서 절대 반복해서는 안 된다. 단, 기존 시스템과의 금액 호환성 검증 시 해당 버그의 영향 범위를 알아야 차이 원인을 추적할 수 있다.

## 3. 본인부담상한제

### 3-1. 개요

환자의 연간 본인부담금 누적액이 상한액을 초과하면, 초과분을 공단이 부담하는 제도이다.

### 3-2. 적용 흐름

```
1. 옵션 확인: M_OverUserPriceYN == "Y"
2. 연간 누적 본인부담금 조회: GetPbRealPrice(환자코드, 조제일자, 환자ID)
3. 연간 상한액 조회: GetConfigValue("P", "P16", 연도)
4. 누적분 + 현재 PbRealPrice → 초과분 계산
5. 초과 시:
   - PbRealPrice < 초과분 → 전액 이관 (PbOverUserPrice = PbRealPrice, PbRealPrice = 0)
   - PbRealPrice >= 초과분 → 일부 이관 (PbRealPrice -= 초과분, PbOverUserPrice = 초과분)
6. PbSumUser -= PbOverUserPrice (환자부담금에서 차감)
```

### 3-3. 핵심 수식

```
pb_RealPrice(DB 누적) + PbRealPrice(현재 처방) = 연간합산
연간합산 - 연간상한액 = 초과분 (양수일 때만)

PbOverUserPrice = min(초과분, PbRealPrice)  // 현재 처방에서 이관 가능한 금액
PbRealPrice -= PbOverUserPrice               // 환자 실부담 차감
PbSumUser -= PbOverUserPrice                 // 환자부담금 총액 차감
```

### 3-4. 구현 주의사항

- 연간 상한액(P16)은 매년 변동 가능하며, 소득 구간별로 다를 수 있다.
- `GetPbRealPrice`는 해당 환자의 당해연도 누적 본인부담금을 반환한다.
- 본인부담상한제 적용 후에도 `m_TDay_Price`(명절가산)는 별도로 `PbInsuPrice`, `PbTotalPrice`, `PbSumInsure`에 추가된다.

## 4. 특수약품 하드코딩: 648903860

### 4-1. 약품 개요

- **약품코드**: 648903860
- **약품명**: 코로나19 경구 치료제 (코로나19 치료제 3종 중 하나)
- **코로나19 치료제 전체**: 648903670, 655502130, 648903860

### 4-2. 투약일수 제한 (5일 상한)

```
조건: 약품코드 == "648903860" AND 급여(PD_INSUPAY=1) 약품 존재
처리: 투약일수 = min(처방투약일수, 5)

648903860의 투약일수(num7) 계산 시:
  - num(급여 내복약 최대일수) >= 5 → 5
  - num == 0 → 1
  - 그 외 → num
```

### 4-3. 5% 가산 (2024.10.25~)

2024년 10월 25일부터, 648903860 약품의 약가 합계에 대해 본인부담금 산정 시 5% 가산이 적용된다.

```
조건: 조제일자 >= 20241025 AND 648903860 급여약품 합계(num9) > 0

보훈코드별 분기:
  - M61, M20: 가산 적용 (num10 = num9 * 0.05)
  - M10, M83, M82: 가산 제외 (num10 = 0)
  - 기본(보훈 없음 포함): 가산 적용 (num10 = num9 * 0.05)
```

**본인부담금 산정 시**: 648903860 약가를 별도 분리하여 계산한다.

```
일반 산정:
  PbUserPrice = trunc100((총액 - 648903860약가) * 요율%) + 648903860약가 * 5%

V252(중증질환) 산정:
  PbUserPrice = trunc10((총액 - 648903860약가) * V252요율%) + 648903860약가 * 5%
```

### 4-4. 구현 주의사항

- 648903860은 코로나19 치료제이므로, 보험 여부와 무관하게 **급여 약품으로 취급**된다.
- 5% 가산은 2024.10.25 이후에만 적용되며, 이전에는 일반 요율로 계산한다.
- 보훈 M10(국비100%), M83(90%), M82(0%)는 가산에서 제외된다.

## 7. 비급여 반올림 옵션 (NPayRoundType 6종)

### 7-1. 옵션 목록

| NPayRoundType | 의미 | 절사/반올림 단위 | 적용 방식 |
|---------------|------|-----------------|-----------|
| `""` (빈값) | 기본 (반올림 없음) | - | 비급여 금액 별도 처리 안함 |
| `"N"` | 미적용 | - | 비급여 반올림 하지 않음 |
| `"10_U"` | 10원 단위 올림 | 10원 | `ceil(금액 / 10) * 10` |
| `"10_M"` | 10원 단위 반올림 | 10원 | `round(금액 / 10) * 10` |
| `"10_D"` | 10원 단위 내림 | 10원 | `floor(금액 / 10) * 10` |
| `"100_U"` | 100원 단위 올림 | 100원 | `ceil(금액 / 100) * 100` |
| `"100_M"` | 100원 단위 반올림 | 100원 | `round(금액 / 100) * 100` |

### 7-2. 적용 조건

```
조건: 보훈코드 없음 AND M_Ps_Special_Pub != "101"
대상: G보험(자보) 또는 비급여 약품 합산금액
```

### 7-3. 적용 범위

- **G보험**: `PbSumWageComm + SumUserDrug - Pub100Price` 전체에 적용
- **비G보험**: `SumUserDrug_100(U항)` 포함/제외 여부에 따라 차등 적용

### 7-4. 차액 보정

`NPayRoundType != "N"`일 때, 반올림 전후 차액이 발생하면 이를 보정한다.

```
G보험: 차액 = 반올림 전 금액 - 반올림 후 금액
비G보험: NPayRoundType에 따라 Pub100Price 포함 여부를 결정하여 차액 계산
```

### 7-5. 구현 주의사항

- 이 옵션은 약국별 환경설정(`PrsBillCalcOptions.NPayRoundType`)으로 전달된다.
- 보훈 환자에게는 적용하지 않는다.
- `NPay_Round_F10_YN = "Y"`이면 F10 보험도 G보험처럼 비급여 처리한다.

## 8. 100%본인부담(U항) 처리

### 8-1. 공단 PDF 규정 (p.648~655)

#### 기본 원칙
- 전액(100/100) 본인부담 약제는 명세서 **U항**에 기재한다.
- 처방내역의 '본인부담률 구분코드'란에 **'U'**를 반드시 기재한다.
- 약국은 처방내역과 조제투약내역을 **모두** 기재한다.

#### 조제료 산정
- **조제료, 의약품관리료 등 관련 수가는 요양급여비용으로 산정**한다.
- 즉, U항 약제의 조제료는 일반 급여(01/02항)로 청구한다.
- 약제비(약가)만 100% 본인부담이다.

#### 금액 집계
- U항 금액 → `건강보험(의료급여)100분의100본인부담금총액`으로 집계
- U항 금액은 `요양급여비용총액1`에 **포함하지 않음**

### 8-2. 동일 약제의 급여/U항 분리

동일 약제가 일부는 보험급여, 일부는 100% 본인부담인 경우:
- 처방내역: 급여분과 U분을 줄 분리하여 각각 기재 (급여분은 빈값/A/B/D/E, U분은 'U')
- 조제투약내역: 급여분은 01항, U분은 U항에 각각 기재

**예시** (소마지나정 30일분 중 15일분 급여 + 15일분 U항):
```
[처방내역]
642201210 소마지나정 1 1 15      (급여)
642201210 소마지나정 1 1 15 U    (100% 본인부담)

[조제투약내역]
01항: 642201210 1,106원 × 1 × 1 × 15 = 16,590원   (급여)
02항: Z1000, Z2000, Z3000, Z4326, Z5000              (조제료 - 전체 급여)
U항:  642201210 1,106원 × 1 × 1 × 15 = 16,590원   (100% 본인부담)
```

### 8-3. EDB/유팜 구현 방식

- EDB: `PD_INSUPAY` 값으로 급여유형 분류, `m_pb_SumInsuDrug_100`에 U항 약가 누적
- 유팜: `전액본인부담` 플래그로 분류, 조제료 계산 시 급여로 취급

### 8-4. 구현 주의사항

- U항 약가는 요양급여비용총액1에서 **제외**하되, 요양급여비용총액2에는 **포함**한다.
- 조제료 산정 시 U항 약품의 투약일수도 급여 약품과 동일하게 반영한다.
- NPayRoundType 적용 시 `Pub100Price(U항)`의 포함/제외를 정확히 분기해야 한다.

## 9. 비즈팜 고유 버그 (새 시스템에서 반복 금지)

### 9-1. BUG-001: Dead Code (intZBaseAmt 자기비교)

| 항목 | 내용 |
|------|------|
| **위치** | `Ppre1000__.frm` 라인 9825 |
| **코드** | `If intZBaseAmt <> intZBaseAmt Then` |
| **문제** | Long 타입 자기비교는 항상 False → 내부 블록 미실행 |
| **영향** | 보훈 환자 + 정액 구간에서 `Insur_Amt - Bohun_Amt <= Insur_Self_Amt`일 때, 본인부담금이 조정되지 않아 과다 청구 발생 |
| **심각도** | HIGH (P1) |
| **빈도** | 낮음 (보훈+정액 특정 조건) |

**새 시스템 대응**: 보훈 환자에서 `insurAmt - bohunAmt <= insurSelfAmt`인 경우 본인부담금 재조정 로직을 반드시 구현한다.

```csharp
// 수정 방향
if (insurAmt - bohunAmt > insurSelfAmt)
{
    chungAmt = insurAmt - insurSelfAmt - eHealthAmt;
}
else if (bohunAmt > 0)
{
    insurSelfAmt = Math.Max(0m, insurAmt - bohunAmt);
    chungAmt = Math.Max(0m, bohunAmt - eHealthAmt);
}
```

### 9-2. BUG-002: 이중등호 (50%감면 + 암환자)

| 항목 | 내용 |
|------|------|
| **위치** | `Ppre1000__.frm` 라인 9732 |
| **코드** | `Insur_Self_Amt = Insur_Self_Amt = Int((...))` |
| **문제** | VB6에서 `A = B = C`는 B와 C를 비교한 Boolean(-1/0)을 A에 대입 |
| **영향** | 암환자(chkCa=1) + 50%감면 + 조합/보험 가입자의 본인부담금이 0원 또는 -1원이 됨 |
| **심각도** | **CRITICAL (P0)** |
| **빈도** | 중간 (해당 조건 충족 시 항상 발생) |

**새 시스템 대응**: 50%감면 본인부담금을 정상 산식으로 구현한다.

```csharp
// 60%감면: insurSelfAmt = floor(insurAmt * 40% * 0.1) * 10 * 10 / 100
// 50%감면: insurSelfAmt = floor(insurAmt * 50% * 0.1) * 10 * 10 / 100  ← 수정
// 30%감면: insurSelfAmt = floor(insurAmt * 70% * 0.1) * 10 * 10 / 100
```

### 9-3. Z4121 Holiday_gb 7/8 교차

| 항목 | 내용 |
|------|------|
| **위치** | `Ppre1000__.frm` 라인 9556~9574 |
| **문제** | Z4121(내외용동시)의 Holiday_gb 매핑이 다른 Z코드와 **반대** |

**다른 Z코드 (Z4xxx, Z4120 등)의 매핑**:
```
Holiday_gb "1" (야간) → *010 접미사
Holiday_gb "8" (6세+야간) → *010 접미사
Holiday_gb "5" (공휴일) → *050 접미사
Holiday_gb "7" (6세+공휴일) → *050 접미사
```

**Z4121만의 매핑 (교차)**:
```
Holiday_gb "1" (야간)     + "7" (6세+공휴일) → Z4121010
Holiday_gb "5" (공휴일)   + "8" (6세+야간)   → Z4121050
```

이는 6세+공휴일("7")이 야간 코드("010")에, 6세+야간("8")이 공휴일 코드("050")에 매핑되는 교차 오류다. 의도적 설계인지 버그인지 판단이 필요하며, PZCVALUE DB에 해당 코드가 실제로 존재하는지 확인해야 한다.

**새 시스템 대응**: 일관된 매핑 체계로 통일하되, 기존 DB 수가표와의 호환성을 반드시 확인한다.
