# CH00: 기준 데이터 입력 규격

> 이 문서는 약제비 계산 엔진을 개발하기 위해 **어떤 데이터가 필요한지** 정의합니다.
> 4개 소스(비즈팜/공단/유팜/EDB) + 기존 분석(docs/)을 합집합으로 병합하고, 감독관 검수 + 검증전문가 팩트체크를 거쳤습니다.

## 1. 계산 엔진 입력 파라미터 (처방전/조제건 단위)

## 2. 약품 1건 데이터 (DrugItem)

처방전에 포함된 약품 하나의 정보입니다.

### 2.1 핵심 필드

| 필드 | 타입 | 의미 | 출처 |
|------|------|------|------|
| 약품코드 | string (9자리) | EDI코드 (보험등재약) | [공단][비즈팜][유팜][EDB] |
| 복용구분 | string | 내복/외용/주사 | [전체] |
| 급여구분 | string | 급여/비급여/부분부담/100%본인 | [전체] |
| 1회투약량 | decimal | 1회 투여량 (소수 4자리) | [공단][EDB] |
| 1일투여횟수 | decimal | 1일 투여 횟수 (소수 2자리) | [공단][EDB] |
| 총투여일수 | decimal | 총 투여 일수 | [전체] |
| 단가 | decimal | 약품 단가 (원) | [전체] |
| 포장단위 | decimal | 팩 수량 (0이면 미적용) | [유팜][EDB] |

### 2.2 분류/속성 필드

| 필드 | 타입 | 의미 | 출처 |
|------|------|------|------|
| 산제여부 | string | 1=산제(가루약) | [유팜][EDB] |
| 마약향정구분 | string | 3=마약, 4=향정 | [EDB] |
| 제외유형 | string | EXTYPE (1/9 등) | [EDB] |
| 성분코드 | string | 약품 성분 | [EDB] |
| 보험약품여부 | string | 1=보험등재 | [EDB] |

### 2.3 금액 자동계산 공식

```
소모량 = 1회투약량 × 1일투여횟수 × 총투여일수
if (포장단위 > 0) 소모량 /= 포장단위
약품금액 = (int)(소모량 × 단가 + 0.5)   // 사사오입 (원미만)
```

> **검증 결과**: 비즈팜 `Int(x+0.5)`, EDB `(int)(x+0.5m)`, 유팜 `To원미만사사오입`, 공단 "원미만 4사5입" — **양수 영역에서 4소스 모두 동일 동작** 확인됨.

### 2.4 코드 매핑 (소스별 차이)

**복용구분:**

| 구분 | 공단 (목번호) | EDB (PD_TAKE) | 비즈팜 (Jaeje_Gb) |
|------|:---:|:---:|:---:|
| 내복약 | 01 | 0 | 1,7,8,9 |
| 외용 | 02 | 1 | 2 등 (기타) |
| 주사 | 03 | 2 | 3 |

> **검증 결과**: 공단(1-based)과 EDB(0-based)는 순서 동일, offset +1 차이만 존재.

**급여구분:**

| 구분 | 공단 (항번호) | EDB (PD_INSUPAY) |
|------|:---:|:---:|
| 약가 (급여) | 01 | 1 |
| 조제료 등 | 02 | (해당없음) |
| 선별급여 50% | A | 2 |
| 선별급여 80% | B | 3 |
| 선별급여 30% | D | 4 |
| 선별급여 90% | E | 5 |
| 100% 본인부담 | U | 9 |
| 보훈 100% | V | (별도처리) |
| 비급여 | W | 0 |

## 3. 수가 마스터 데이터

Z코드별 조제료 단가를 관리하는 마스터 테이블입니다.

### 3.1 구조

| 항목 | 비즈팜 | 유팜 | EDB |
|------|--------|------|-----|
| 테이블 | PZCVALUE | ST조제료 + ST조제료적용일 | EP_BASES.DBO.MEDIWAGE |
| 저장 단위 | 원(단가) | 점수 + 기준금액 | 원(단가) |
| 조회 키 | drug_id + 처방일 | 적용일ID + 코드 | DueDate(MAX≤처방일) |

> **감독관 식별 불일치 #4**: 유팜만 "점수 기반"이고 비즈팜/EDB는 "단가 기반". 계산 결과는 동일하나 저장 방식이 다름.
> **검증 결과**: 점수×환산지수 → 원단위 사사오입 → 10원미만 사사오입 = 최종 단가. EDB Z1000=790원은 7.45×105.5=786.475 → 786 → **10원 사사오입 → 790원**으로 정확함.

## 7. 감독관 식별 주요 불일치 + 검증 결과

| # | 불일치 내용 | 판정 | 통합 방침 |
|---|-----------|------|----------|
| 1 | 복용구분 코드 (0/1/2 vs 01/02/03) | 순서 동일, offset만 다름 | 내부는 0-based, 청구서 출력 시 +1 변환 |
| 2 | 급여구분 코드 (숫자 vs 알파벳) | 내부코드 vs 청구코드 | 매핑 테이블 필수 |
| 3 | 보험유형 코드 (4종 체계) | 설계 차이 | 대분류(C/D/G/F/E) + 세부속성 조합 방식 채택 |
| 4 | 수가 저장 (점수 vs 단가) | 결과 동일 | 단가 기반 채택 (점수→단가 변환은 마스터 갱신 시) |
| 5 | 정액 기준금액 (8,000 vs 10,000) | 비즈팜은 구 기본값 | **10,000원** (공단 현행, DB 관리) |
| 6 | 약품금액 반올림 | 4소스 동일 | `(int)(x + 0.5m)` = 사사오입 |
| 7 | 수가 단가 변환 | 10원 사사오입 포함 | 점수×환산지수 → 원 사사오입 → **10원 사사오입** |
| 8 | 정액 본인부담금 (1,000 vs 1,500) | 연도 차이 가능성 | DB 파라미터로 관리 (시점별) |

## 8. 다른 AI에게 전달할 핵심 메시지

1. **모든 기준금액/요율/정액은 DB 테이블에서 조회**해야 합니다. 코드에 하드코딩하면 수가 개정 시 대응 불가.
2. **반올림 규칙이 단계마다 다릅니다**: 약품금액(원미만 사사오입) → 수가단가(10원미만 사사오입) → 총액(10원 절사) → 본인부담(100원 절사).
3. **보험유형은 "대분류 + 세부속성" 조합**으로 설계하세요. 평탄화된 단일 코드는 조합 폭발을 유발합니다.
4. **입력 정밀도**: 1회투약량 소수 4자리, 1일투여횟수 소수 2자리, 금액은 decimal 타입 필수.
5. **코드 매핑 테이블**이 필요합니다: 내부코드(0-based) ↔ 청구서코드(1-based/알파벳).
